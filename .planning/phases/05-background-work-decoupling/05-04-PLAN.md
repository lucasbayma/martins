ROADMAP.md Phase 5 success criteria (what must be TRUE — user judges by feel):

1. The 5s diff-refresh timer is gone — diff refresh only fires on actual file-system events (or the 30s safety-net fallback).

4. Creating, archiving, or deleting a workspace feels instant — the state-save to `~/.martins/state.json` never produces a visible pause.
5. Sitting in the app for several minutes, the user experiences no random lag spikes — interaction feels consistent, not "sometimes fine, sometimes stuck".

### UAT 1 — BG-01: The 5s diff-refresh timer is gone

**Setup:** Open martins with a workspace pointing at a git project that has some uncommitted modifications (so the diff list on the right sidebar is populated). Open Activity Monitor (or `top -pid $(pgrep -f martins)` in a second terminal) and watch the martins process CPU.

**Observe:** Sit idle for 60+ seconds. Do NOT edit any files, do NOT touch the keyboard or mouse.

**Expected:** CPU stays at near-zero (<2%) the entire time. There is NO periodic 5s spike to ~9% CPU that was present before Phase 5.

**Acceptable variation:** At t≈0 and t≈30s, a small (<1% for <200ms) tick may appear — this is the safety-net refresh_tick firing and dispatching a git2 run on a blocking worker. The tick should be brief and barely visible.

**Record in PHASE-SUMMARY:** PASS if no 5s-period spike. FAIL if the old 5s spike pattern is still present.

### UAT 2 — BG-02 + BG-01: External editor edit → diff view updates ~200ms

**Setup:** With martins running + a workspace open, use an external editor (vim/nano/emacs in another terminal, or VS Code, or any file-editing tool of choice) to edit a file inside the active worktree. The file should be tracked (not `.gitignore`d, not inside `.git/` or `target/` etc.).

**Observe:** Save the file in the external editor. Immediately glance at martins's right sidebar.

**Expected:** Within ~200-400ms of the save, the diff view updates to reflect the change (either a new file appears in the modified list, or an existing file's diff changes). NO visible stall in the TUI during this time.

**Acceptable variation:** First save after a long idle may take slightly longer (FSEvents warmup on macOS). Subsequent saves should feel instant.

**Record in PHASE-SUMMARY:** PASS if update feels sub-400ms. FAIL if update takes >1s OR if the TUI stalls during the update.

### UAT 3 — BG-04: Burst file changes → ≤1 refresh (coalescing)

**Setup:** In the same worktree, open a terminal at the worktree root. Run a burst-producing command:
```bash
# Option A — cargo build (produces many file changes in target/)
cargo check
# Option B — git checkout of a branch (produces many file changes)
git stash && git stash pop
```

**Observe:** Watch the right sidebar during and immediately after the burst.

**Expected:** The diff view does NOT flicker through multiple intermediate states. At most one refresh visible after the burst completes. For `cargo check`, files in `target/` must NOT trigger refreshes (the `is_noise` filter in watcher.rs excludes `target/`).

**Acceptable variation:** If the burst crosses a debounce window boundary (>200ms), you may see 2 refreshes — this matches the `count <= 2` assertion in `debounce_rapid_burst_of_10`.

**Record in PHASE-SUMMARY:** PASS if ≤2 refreshes observed AND no stall during the burst. FAIL if the TUI freezes during the burst OR if `target/` files trigger a refresh.

### UAT 4 — BG-05: Workspace create/archive/delete feels instant

**Setup:** In martins, perform the following sequence:
1. Create a new workspace (press the shortcut for new-workspace or click the + button if present). Fill the form, confirm.
2. Archive the new workspace you just created (select it, trigger the archive flow, confirm).
3. Delete the archived workspace (navigate to archived state, trigger delete, confirm).

**Observe:** After each Enter/confirm, note whether the TUI pauses. Pay specific attention to:
- The modal confirm → view update transition (must feel instant — the save_state_spawn runs in the background).
- The archive confirm → sidebar refresh transition (must feel instant — `remove_dir_all` now runs in a spawn_blocking task).

**Expected:** Each action feels instant. No visible pause between the Enter/click and the view updating.

**Acceptable variation:** If you create a workspace from a very large repo (>10GB), initial tab setup (tmux new-session + PTY spawn) may briefly flash — that's expected and out of scope for Phase 5 (RESEARCH §17 Q4 defers tmux ops).

**Record in PHASE-SUMMARY:** PASS if no visible pause on confirm. FAIL if the TUI freezes for >100ms after confirm.

### UAT 5 — BG-05 / general: Sit for several minutes, no random lag spikes

**Setup:** Leave martins running with a workspace + tab open. Use it normally for 5+ minutes: type into the PTY, switch tabs, click sidebar items, edit files externally, etc.

**Observe:** Note any moments where the app "hitches" — feels stuck for a few hundred ms before responding.

**Expected:** No random lag spikes. Interaction feels consistently responsive throughout the session.

**Acceptable variation:** A brief hitch immediately after a workspace-archive on a node_modules-heavy worktree is acceptable — `remove_dir_all` can generate many FSEvents bursts that the watcher debouncer coalesces. The hitch, if any, should be <200ms and recoverable.

**Record in PHASE-SUMMARY:** PASS if no perceptible random lag spikes over 5+ minutes. FAIL if you experience ≥1 random hitch you can't attribute to a specific action.
